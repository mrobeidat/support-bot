1. AI Agents

What is an AI Agent?

An AI agent is an LLM in a loop that can take actions. Unlike a simple chatbot that just generates text responses, an agent can:

  • Reason about what information it needs
  • Act by calling tools (APIs, databases, functions)
  • Observe the results and decide what to do next
  • Repeat until the task is complete

This is called the ReAct pattern (Reasoning + Acting).

User: "I want to return order #12345" | v ┌─────────────────────────────────────┐ │ LLM receives message + tool list │ │ Thinks: "I need to check the │ │ order status first" │ └──────────────┬──────────────────────┘ │ v ┌─────────────────────────────────────┐ │ Tool Call: check_order("ORD-12345")│ ← Agent DECIDES to act └──────────────┬──────────────────────┘ │ v ┌─────────────────────────────────────┐ │ Tool Result: {status: "delivered", │ ← Your code EXECUTES the tool │ items: ["Keyboard"], ...} │ └──────────────┬──────────────────────┘ │ v ┌─────────────────────────────────────┐ │ LLM receives result, thinks: │ │ "Order is delivered, eligible for │ │ return. OpenCX Connection: OpenCX's AI agents do exactly this — they autonomously handle customer queries by searching knowledge bases, checking order info, creating tickets, and escalating to humans. As a Customer Engineer, you'd help customers configure and debug these agent behaviors.

Key Components of an Agent

1. LLM (The Brain)

The language model that does the reasoning. Claude, GPT-4, etc. It reads the conversation, decides what to do, and generates responses.

2. Tools (The Hands)

Functions the agent can call to interact with the outside world — APIs, databases, search engines. The LLM decides when and which tool to use.

3. System Prompt (The Instructions)

Defines the agent's role, rules, personality, and behavior boundaries. It's the "employee handbook" for the AI.

4. The Loop (The Workflow)

The orchestration code that sends messages to the LLM, executes tool calls, feeds results back, and repeats until done. Includes safety measures like max iterations.

5. Memory / Context

The conversation history that lets the agent remember what was said earlier. In longer conversations, you might need to summarize to fit within the context window.

2. RAG (Retrieval Augmented Generation)

What is RAG?

RAG is a technique that gives an LLM access to external knowledge at query time, without fine-tuning the model. Instead of hoping the LLM knows the answer, you retrieve relevant documents and inject them into the prompt.

Think of it like: Instead of memorizing an entire textbook, you look up the relevant pages before answering a question.

Why RAG? (vs alternatives)

ApproachProsCons
Fine-tuningDeeply learned knowledgeExpensive, goes stale, can hallucinate, hard to update
Stuff everything in promptSimpleHits context window limit fast, expensive per-call
RAGFresh data, cheap, easy to update, reduces hallucinationRetrieval quality matters, needs embedding pipeline

RAG Pipeline

INDEXING (one-time, when docs change): Documents ──→ Chunk ──→ Embed ──→ Store (markdown, (split (convert (save vectors PDFs, into text to in a vector web pages) sections) vectors) database) QUERYING (every user question): User Question ──→ Embed ──→ Search ──→ Top-K Results ──→ Inject into Prompt ──→ LLM Answer (same (cosine (most relevant (add as context) model) similarity) chunks)

Key Concepts

Chunking

Splitting documents into smaller pieces. Strategies:

  • By heading — split on ## headers (what we use in our bot)
  • By paragraph — split on double newlines
  • Fixed size — split every N tokens with overlap
  • Semantic — use AI to find natural topic boundaries

Tradeoff: Too small = chunks lack context. Too large = chunks contain noise and waste the context window.

Embeddings

Converting text into a vector (array of numbers) that captures its meaning. Similar texts produce similar vectors.

Example: "shipping policy" and "delivery times" would have similar embeddings, even though they share no words.

Models: OpenAI text-embedding-3-small, Cohere embed-v3, open-source all-MiniLM-L6-v2

Vector Search (Cosine Similarity)

Finding the most similar vectors by measuring the angle between them. Score ranges from -1 to 1, where 1 = identical direction.

similarity = dot(A, B) / (magnitude(A) * magnitude(B))

This is what makes semantic search work — it matches by meaning, not just keywords.

Re-ranking (Advanced)

After initial retrieval, use a more expensive model to re-score the results for relevance. Improves accuracy but adds latency.

3. Tools / Actions (Function Calling)

What are Tools?

Tools are functions that an LLM can choose to call during a conversation. The LLM doesn't execute them — it outputs a structured request, and your code runs the actual function.

How Function Calling Works

1. YOU define tools as JSON schemas: { "name": "check_order", "description": "Look up order status by ID", "parameters": { "type": "object", "properties": { "order_id": { "type": "string" } }, "required": ["order_id"] } } 2. YOU send the schemas alongside the conversation to the LLM 3. LLM DECIDES to call a tool (instead of responding with text): { "tool_use": { "name": "check_order", "arguments": { "order_id": "ORD-12345" } } } 4. YOUR CODE executes the function and returns the result 5. LLM receives the result and continues reasoning

Tool Design Best Practices

Clear Descriptions Matter

The LLM reads the tool description to decide when to use it. Vague descriptions = wrong tool usage.

Bad: "description": "order stuff"

Good: "description": "Look up the status of a customer order by order ID. Returns order details including status, items, tracking number, and delivery estimate."

Error Messages Should Help the LLM Recover

Bad: { "error": "not found" }

Good: { "error": "No order found with ID X. Please verify the order number with the customer." }

The LLM reads the error and can ask the customer for the correct info.

Security: Don't Expose Destructive Actions Without Guardrails

A tool like delete_account() needs confirmation flows. The LLM might call it prematurely. Add a confirm parameter or a separate confirmation step.

4. System Prompts & Prompt Engineering

What is a System Prompt?

The instruction manual for the AI agent. It defines who the agent is, what it should do, how it should behave, and what it should never do. Everything the agent does is influenced by the system prompt.

Anatomy of a Great System Prompt

1. Role & Persona

"You are a friendly and professional customer support agent for TechStore, which sells electronics and accessories."

Sets the identity. The agent will adopt this persona in all responses.

2. Knowledge Boundaries

"ONLY answer questions using information from the knowledge base (search_knowledge_base tool). Do NOT make up information."

This is the #1 way to reduce hallucination. Tell the agent exactly where its knowledge comes from.

3. Tool Usage Rules

"Always check order status before answering shipping questions."
"Use escalate_to_human when the customer is upset or explicitly asks for a person."

Explicit rules for when to use which tool. Without these, the agent guesses.

4. Tone & Style

"Be friendly, professional, and concise. Use bullet points for lists."

Controls the brand voice. A B2B enterprise wants formal; a D2C startup wants casual.

5. Escalation Rules

"Transfer to a human if: customer is angry, issue involves billing, or you cannot resolve in 3 messages."

Critical for customer support — knowing when to hand off.

6. Few-Shot Examples (optional)

Example:
Customer: "Where's my order?"
Agent: "I'd be happy to help! Could you share your order number so I can look it up?"

Show the agent ideal behavior for tricky cases.

Practice Q&A: AI Agents

▶ What is an AI agent and how is it different from a chatbot? Basic

Answer: An AI agent is an LLM combined with tools and a reasoning loop. Unlike a chatbot that can only generate text responses, an agent can take actions — it calls external tools, gathers information, and makes multi-step decisions before responding.

The key difference is autonomy. A chatbot responds directly to each message. An agent might receive "I want to return order #12345" and autonomously: (1) look up the order, (2) check if it's eligible for return, (3) search the return policy, and (4) provide specific instructions — all without being explicitly told to do each step.

▶ Walk me through how an agent loop works in code. Intermediate

Answer:

  1. User sends a message
  2. We send the conversation history + available tools + system prompt to the LLM
  3. The LLM responds with either text (done) or a tool call
  4. If it's a tool call, our code executes the tool and feeds the result back to the LLM
  5. We go back to step 2 — the LLM can make another tool call or finally respond
  6. We have a safety limit (e.g., 10 iterations) to prevent infinite loops
while (iteration < MAX_ITERATIONS) {
  response = await llm.chat(messages, tools, systemPrompt)
  if (response.stopReason === "end_turn") return response.text
  if (response.stopReason === "tool_use") {
    result = await executeTool(response.toolCall)
    messages.push(toolResult)  // feed back to LLM
  }
}

▶ How do you handle errors in an agent loop? What if a tool fails? Intermediate

Answer: Several layers of error handling:

  • Max iterations — prevent infinite loops with a hard limit (e.g., 10 steps)
  • Tool execution errors — catch errors and return a helpful message to the LLM so it can recover (e.g., "Order not found, please verify the order number")
  • LLM errors — handle API failures with retries and exponential backoff
  • Graceful fallback — if all else fails, escalate to a human agent rather than showing an error

The key insight is that error messages should help the LLM recover, not just say "error". If a tool returns "Order not found — ask the customer to verify their order number", the LLM can do exactly that.

▶ What are the main challenges in building production AI agents? Advanced

Answer:

  • Reliability — agents can go off-track, call wrong tools, or loop. Need monitoring and guardrails.
  • Latency — each tool call adds an LLM round-trip. Multi-step tasks can be slow. Solution: parallelize independent tool calls.
  • Cost — every iteration costs API tokens. Long conversations with many tool calls get expensive.
  • Evaluation — hard to test because behavior is non-deterministic. Need automated evals + human review.
  • Context window management — long conversations overflow. Need summarization strategies.
  • Security — prompt injection attacks where user input tricks the agent into misusing tools.

Practice Q&A: RAG

▶ What is RAG and why would you use it instead of fine-tuning? Basic

Answer: RAG (Retrieval Augmented Generation) gives an LLM access to external knowledge by retrieving relevant documents and injecting them into the prompt at query time.

Why not fine-tuning?

  • Cost — fine-tuning is expensive; RAG just needs an embedding model
  • Freshness — when docs change, just re-index. Fine-tuning requires retraining
  • Hallucination — RAG provides explicit source text, reducing made-up answers
  • Transparency — you can show users which documents the answer came from (citations)
  • Speed to deploy — RAG can be set up in hours; fine-tuning takes days
For OpenCX: customers upload their help articles and expect the agent to answer from them immediately — RAG makes this possible without retraining any model.
▶ Walk me through the RAG pipeline step by step. Basic

Answer — two phases:

Indexing (one-time):

  1. Ingest — load documents (PDFs, markdown, web pages)
  2. Chunk — split into smaller, meaningful pieces (by heading, paragraph, or fixed size)
  3. Embed — convert each chunk into a vector using an embedding model
  4. Store — save vectors in a vector database (Pinecone, ChromaDB, etc.)

Querying (every user question):

  1. Embed the question — same embedding model as indexing
  2. Search — find the most similar vectors (cosine similarity)
  3. Retrieve top-K — get the 3-5 most relevant chunks
  4. Generate — inject chunks into the prompt as context, LLM answers grounded in them
Draw this as a diagram if on a whiteboard. Mention that you implemented this from scratch — chunking, embedding, cosine similarity, and retrieval.
▶ What are the tradeoffs in chunk size? Intermediate

Answer:

  • Too small (1-2 sentences) — chunks lack context, you might retrieve a fragment that doesn't make sense alone
  • Too large (entire pages) — chunks contain noise, relevant info is diluted, wastes context window space
  • Sweet spot — usually a paragraph or a section under one heading (200-500 tokens)

Practical approach: Start with heading-based chunking for structured docs (like help articles), paragraph-based for unstructured text. Use overlap (include the last sentence of the previous chunk) to maintain continuity.

Mention that in your bot, you chunk by ## heading which works well for help center articles — each section is a self-contained topic.
▶ What is cosine similarity and why do we use it for vector search? Intermediate

Answer: Cosine similarity measures the angle between two vectors, not their magnitude. It returns a value from -1 to 1, where 1 means identical direction.

Why cosine over Euclidean distance? Because we care about the direction of meaning, not the length of the vector. A longer document and a shorter one about the same topic should be "similar" — cosine handles this because it normalizes for length.

Formula: cos(A, B) = (A · B) / (|A| × |B|)

In practice, you compute dot products between the query vector and all stored vectors, then return the top-K highest scores.

▶ A customer says the AI agent gives wrong answers. How do you debug RAG? Advanced

Answer — debug the chain step by step:

  1. Is the document in the knowledge base? Maybe the info was never uploaded
  2. Is it chunked correctly? Maybe the relevant info was split across two chunks and neither is complete enough
  3. Is it being retrieved? Run the query manually and check which chunks come back. Maybe the embedding similarity is too low
  4. Is the right chunk ranked high enough? Maybe it's result #6 but you only use top 3
  5. Is the LLM using the context correctly? Maybe the chunk is retrieved but the prompt doesn't instruct the LLM to follow it

Common fixes:

  • Restructure the knowledge base article so each section is self-contained
  • Increase top-K from 3 to 5
  • Add keyword search alongside semantic search (hybrid search)
  • Add the article title to each chunk for context (we do this in our bot)
This is the exact kind of debugging you'd do as a CE at OpenCX — walk the customer through why the agent gave a wrong answer and how to fix it.

Practice Q&A: Tools & Actions

▶ How does function calling / tool use work? Basic

Answer:

  1. You define tools as JSON schemas — name, description, and parameters with types
  2. You send these schemas to the LLM alongside the conversation
  3. The LLM reads the descriptions and decides if it needs to call a tool
  4. If yes, it returns a tool_use response with the tool name and arguments (instead of text)
  5. Your code executes the actual function — the LLM never runs anything itself
  6. You feed the result back to the LLM, and it continues reasoning

Key insight: The LLM is the decision-maker, but your code is the executor. This is a security boundary — you control what actually runs.

▶ How do you design good tool schemas? Intermediate

Answer:

  • Clear descriptions — the LLM reads these to decide which tool to use. Be specific about what the tool does and when to use it
  • Descriptive parameter names — order_id is better than id
  • Parameter descriptions — include format hints: "The order ID (e.g., ORD-12345)"
  • Use enums — for fixed options like priority: ["low", "medium", "high"]
  • Mark required fields — so the LLM knows what it must provide
  • Helpful error messages — return errors that tell the LLM how to recover
Show your tool definitions from tools/ — they demonstrate all these practices.
▶ What's the difference between Anthropic and OpenAI tool calling? Intermediate

Answer: The concept is the same, but the API format differs:

  • Schema field: Anthropic uses input_schema, OpenAI uses parameters
  • Tool calls: Anthropic returns tool_use content blocks, OpenAI returns tool_calls array
  • Tool results: Anthropic sends them as user messages with tool_result type, OpenAI uses a tool role
  • System prompt: Anthropic has a separate system field, OpenAI uses a system message in the messages array

That's why I built a provider abstraction — same agent logic works with both APIs.

This shows you've worked with both APIs. Mention the providers/ directory in your project.
▶ How do you handle tool call security? Advanced

Answer:

  • Input validation — always validate tool arguments before executing. Don't trust the LLM to provide correct types or safe values
  • Rate limiting — prevent the agent from making too many API calls (cost + abuse)
  • Confirmation for destructive actions — tools that delete data or spend money should require a confirmation step
  • Scope limitation — only expose tools the agent actually needs. Don't give a support agent a "delete database" tool
  • Prompt injection defense — users might try to trick the agent into calling tools maliciously. The system prompt should have guardrails
  • Audit logging — log every tool call for debugging and compliance

▶ How do you reduce hallucination in an AI agent? Intermediate

Answer — multiple layers:

  • System prompt — explicitly say "Only answer from provided context. If unsure, say so."
  • RAG grounding — always retrieve relevant docs before answering. The LLM has real text to reference
  • Knowledge boundaries — tell the agent what topics it can and cannot answer
  • Citation requirements — ask the agent to cite which article it's referencing
  • Temperature — lower temperature (0-0.3) for factual responses, higher for creative tasks
  • Fallback behavior — "If you cannot find the answer, say 'I don't have information about that' and offer to escalate"
This is huge for the CE role — customers complain when the AI makes things up. You'd help them fix this through prompt engineering and KB improvements.
▶ A customer says the AI's tone is too formal. How do you fix it? Intermediate

Answer: Tone is controlled in the system prompt. I'd:

  1. Look at their current system prompt's tone section
  2. Add specific tone guidance: "Use casual, friendly language. Use contractions (you're, we'll). Avoid corporate jargon."
  3. Add few-shot examples showing the desired tone
  4. If the agent uses emojis or is too casual, add constraints
  5. Test with common customer queries and iterate

Key insight: Prompt engineering is empirical, not theoretical. You change the prompt, test it, observe the output, and refine. It usually takes 3-5 iterations to nail the tone.

▶ What is few-shot prompting and when would you use it? Basic

Answer: Few-shot prompting means giving the LLM examples of ideal input/output pairs in the prompt.

When to use:

  • When the desired output format is specific or unusual
  • For edge cases where the agent might otherwise handle things wrong
  • To set the right tone or style by example

Example:

When a customer asks about a discount that doesn't exist:
Customer: "Do you have a 50% off code?"
Agent: "We don't currently have a 50% discount available, but here's
what we do offer: [then list actual discounts from KB]"

Zero-shot = no examples (just instructions). Few-shot = 2-5 examples. More examples = more reliable behavior for that specific pattern.

▶ A customer's AI agent is giving wrong answers about their return policy. Walk me through how you'd debug this. Intermediate

Answer — systematic debugging approach:

  1. Reproduce the issue — ask the same question and see what happens
  2. Check the knowledge base — is the return policy actually in their uploaded docs? Is it up to date?
  3. Check retrieval — what chunks are being retrieved for this query? Is the return policy chunk coming up?
  4. Check chunk quality — is the relevant info split across chunks? Is the chunk too long and diluted?
  5. Check the system prompt — does it instruct the agent to search the KB before answering? Does it say to only use KB data?
  6. Check for conflicting info — maybe multiple docs mention returns with different policies

Common root causes:

  • The article wasn't re-indexed after updates
  • The chunking split the policy across multiple sections
  • The system prompt doesn't force KB-grounded answers
  • The embedding model doesn't understand domain-specific terms

▶ What metrics would you track to know if an AI agent is performing well? Advanced

Answer:

  • Resolution rate — % of conversations resolved without human escalation
  • CSAT (Customer Satisfaction) — post-conversation surveys
  • Hallucination rate — % of responses containing fabricated information
  • Escalation rate — how often the agent hands off to a human (too high = agent not good enough, too low = maybe not escalating when it should)
  • First response time — how fast the agent responds
  • Retrieval accuracy — are the right KB articles being found?
  • Tool call success rate — % of tool calls that succeed vs. fail
  • Conversation length — shorter = more efficient (usually)
  • Cost per conversation — API token usage per session

Practice Q&A: Scenario Questions

▶ A customer is on a call and their AI agent just went down. What do you do? Intermediate

Answer:

  1. Acknowledge immediately — "I see the issue, let me investigate right now"
  2. Check the basics — is it an API outage (check LLM provider status page)? Is it their account (rate limits, billing)?
  3. Check logs — look at the error messages. Authentication error? Timeout? Tool failure?
  4. Mitigate — if it's a provider outage, can we fail over? If it's a config issue, fix it live
  5. Communicate — keep the customer informed: "This is caused by X, I'm fixing it now"
  6. Resolve + document — fix the issue, confirm it's working, document the root cause and fix for future reference
  7. Follow up — after the call, create an internal ticket if it's a systemic issue that the eng team should address
They want to see you're calm under pressure and follow a systematic approach. Don't panic — diagnose, communicate, resolve.
▶ An enterprise customer wants the AI agent to handle 5 different languages. How would you approach this? Advanced

Answer:

  • Language detection — detect the customer's language from their first message
  • System prompt — instruct the agent to respond in the detected language
  • Knowledge base — ideally have KB articles in each language. If not, the LLM can translate on the fly (less reliable for technical terms)
  • Embeddings — multilingual embedding models (like multilingual-e5) can match across languages
  • Testing — test each language separately, especially for domain-specific terms

Challenges: Translation quality for specialized terms, mixed-language conversations, RTL language support in UI, different cultural expectations for support tone.

  • System prompt = the instruction manual that defines agent behavior
  • Embeddings = convert text to vectors that capture meaning
  • Cosine similarity = measure how similar two vectors are (by angle, not magnitude)
  • Chunking = splitting docs into smaller pieces for retrieval
  • Few-shot = giving examples in the prompt to guide behavior
  • Hallucination = LLM making up information that isn't in the source data
  • ReAct = Reason + Act pattern — think, then use a tool, then think again
Common traps to avoid in the interview:
  • Don't say "the AI calls the API" — YOUR CODE calls the API, the AI just decides to
  • Don't confuse RAG with fine-tuning — they solve different problems
  • Don't say embeddings are "AI understanding" — they're mathematical representations of meaning
  • Don't forget safety — always mention max iterations, error handling, input validation
  • Don't be too theoretical — reference your actual project whenever possible
